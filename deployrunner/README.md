# 代码目录结构

代码目录简单介绍,具体见对应目录下 readme 文档

- api: 各种对外开放 api 接口，包装核心接口适配外部 API 协议，如 HTTP rest api
- build: 自动化脚本与镜像构建
- charts: 产物 helm chart 目录
- cmd: 二进制产物命令行工具代码, 程序入口，并负责根据程序运行参数拼装各个模块.
- error/codes: 错误码目录，目录下通过定义各层级目录的 code.json 文件定义错误码. **不允许手工编写其中的代码**
- pkg: 核心代码,目录下各个目录划分各种模块和功能
- pkg/app: 应用安装与部署核心执行模块
- pkg/store: 低层级的数据存储实现
- pkg/utils: 一些未归类未模块,且可复用的小工具
- test: 单元测试数据准备和简易的测试工具
- trait: 核心接口定义(TODO: 分类与拆分)

## 制品命令行相关代码

分析代码自上而下最简单，命令行制品是该项目所有能力的主要入口，因此专门详细介绍。

* **cmd**：各个程序的命令行，也是代码的主要入口，查看代码可自上而下的阅读。很多差异化代码会在此处编写，因此某些代码会显得有些复杂。
  * builder：部署框架的在生产端使用的工具，用于读取研发侧应用定义生成应用安装包的命令行工具。一般以镜像方式提供。
  * server：deploy-installer服务中真正提供安装部署能力的http服务端二进制。项目的主要入口，与主要能力。实施端最主要的安装部署能力的提供者，最终集成到deploy-installer服务的镜像中，并作为服务启动点。
    * 提供了upgrade子命令，用于自身的数据库升级，服务于后续自身的数据升级。在deploy-installer服务中以pod的initContainer方式运行。
    * server主命令启动是会对中断服务进行恢复重新入队、安装执行主线程启动和http服务接口线程启动等
  * installer：以命令行的方式提供部署框架的安装能力。目前在客户现场目前不常使用，偶尔作为后门，命令行执行的任务与服务执行的任务一般是独立的。开发环境的自动化测试部署环境时使用，建议后续自动化部署部分更改为使用接口。最终会集成到deploy-installer服务镜像中作为二进制命令提供。
  * mini-deploy：3.0.0之前的部署后端服务的最小化安装命令行工具，主要功能为安装指定的deploy-installer服务与其配套的最小化页面。3.0.0后架构调整，新增ProtonSystem包，deploy-installer并入其中，由core-installer安装。mini-deploy被core-installer代替，不再使用。
  * core-installer：core-installer安装当前目录下helm_charts内所有chart服务，用于安装protonSystem包。protonSystem包构建位于大包构建，构建时会组合core-installer二进制与chart包。
    * 安装配置大体分为系统配置、应用配置与组件配置，配置来自命令行、proton-cli-conf以及指定的配置文件(用于开放一般配置)。
      * 系统配置基本来自proton-cli-conf如数据库、镜像仓库等。
      * 应用配置来自于命令行参数，以及配置文件中的声明，一般向整个系统统一设计提供。
      * 组件配置除了deploy-installer是为了兼容旧服务，一部分配置由命令行参数控制，其余可以通过配置文件开放配置
    * 提供proxy子命令提供后门服务的页面代理开放，一般用于第一次安装或正常页面无法提供服务时使用。
  * mproxy：提供protonSystem模块的代理能力。由于架构调整，原放在前端的认证鉴权能力被脱壳下沉到该代理中就进行。有以下能力：
    * protonSystem模块到上层应用如Studio模块，其中包含一些鉴权能力。
    * 客户端安到protonSystem进行认证鉴权，再代理转发到ProtonSystem模块内服务。
    * 单实例和多实例模式的认证鉴权方式不同。单实例走认证包的的认证鉴权模式，通过监听认证包注册生成的oauthClient的进行延迟加载。多实例使用自带的多实例认证鉴权模式。
    * 多实例的认证鉴权分为：1. 仅认证不鉴权 2. 认证且按角色鉴权。
    * 相应认证鉴权规则与代理路由规则结合，由传入的配置文件决定。
      * 配置中allRoleCoreRules为仅认证不鉴权，路由转发至ProtonSystem内部。
      * 配置中CoreRules为认证且需要平台管理员权限，路由转发至ProtonSystem内部。
      * 配置中Rules为认证且需要系统统计管理员权限，路由转发至上层应用实例对应服务，如Studio中的deploy-service
    * mproxy在deployrunner代码库中最终产物仅会生成镜像，最终由deploy-mproxy代码仓库管理服务的chart、路由配置以及最终产物chart的构建。构建出新的mproxy镜像后需要提交到deploy-mproxy代码仓库的chart中。
  * upgrade：数据库数据升级命令行，需要配合数据升级内容文件执行，暂不完善，不对外提供。server命令以子命令方式集成。  
  * utils：上述各个命令行程序代码需要使用的通用部分，一般为各种通用命令行参数、基础的配置加载部分。该部分由于服务于多个命令行工具且需要进行差异化区分和统一抽象不同实现实例的构建，会显得较为混乱。主要是数据存储端的抽象类的实例化、单实例、多实例不同实现的实例化和组合等。

## 其他代码结构

熟悉其他代码结构，在出现BUG时可以快速猜想、定位问题和修复，因此大体介绍各个模块代码。

* api/rest：http接口，目录下为安装/升级相关任务的api接口包装，大多是对gin框架基础的参数的读取，以及内部错误到对外错误码的转换。

* **api/rest/protom_component**：单实例下基础资源组件实例以及连接信息操作的实现，主要基于component-manager进行具体实例的操作，再基于下文中proton模块将配置写回proton-cli-conf

* **api/rest/proton_component/info**：多实例下的基础资源组件的实例以及连接信息操作的实现，主要基于component-manager进行具体实例的操作，再基于下文中关系数据库的proton模块，将数据写入关系数据库进行存储。

* **api/proton_component/compose**：多应用任务组合安装的实现，理论上实现部分应该拆分到pkg，但仅为各个模块的简易组合完成的实现，实际大部分为参数转换和错误转换，实际逻辑部分不多，因此未继续进行代码结构优化。

* **trait**：该模块为整个项目的核心抽象模块以及一些统一的实现，其中有些未能完全分割到pkg模块，因此可以视为最核心的抽象代码部分。从C++之类的角度就是头文件。

* error：开发端生成内部错误码以及运行时生产端加载错误码模块。

  * golang错误码的golang变量代码可以通过tools/generate_errors/main.go程序生成。对应程序会遍历指定的目录中code.json文件并生成对应错误码代码，将生成的golang代码输出到与code.json同级的code.go文件中。
    * 即code.json用于各个层级的错误码定义。
    * 其中输出的error_codes.sqlite3文件为错误码的本地数据存储，为了避免错误码定义变更导致错误码唯一值冲突以及修改
    * 输出的code.go是便于代码使用的golang代码，其中定义了错误码映射的golang代码变量的值，本质是错误码的索引
    * 输出生成的code_cache.json是错误码最终形式，结合code.go可以由索引值转为具体错误码(未来可用于做语言资源等行为，目前暂无)。
  * 由于该模块为后续引入，因此该模块中的code.go中的常量值在实际使用中一般为间接使用。

* pkg：各个层级的业务内部模块会在此实现。

* **pkg/app**：安装/升级模块核心实现。

  * builder：服务于开发侧通过应用定义生成应用安装包的构建模块。会根据应用定义解析应用定义，完成如组件chart下载，最终应用包大包等功能。与cmd/builder相对应
  * executor：服务于实施现场上层应用的安装/升级操作。其通过抽象的关系数据存储store将安装任务以及过程中的执行结果存储于store中。通过pkg/graph模块结合应用定义，生成任务执行计划，并利用pkg/graph提供的任务执行器实现完成各类任务执行。例如helm/service代表的是一般的helm3安装/更新任务
    * executor大体的逻辑为http服务线程作为生产端，执行器为消费端，通过内存内部的队列实现任务的传递。生产端写入队列，执行器从队列中消费并将任务转换执行计划，最终并发的执行各个子任务。
    * executor代码中有大量的并发控制、锁控制以及错误处理，因此代码较为复杂。
    * executor中的复杂核心为基于锁的并发控制。利用数据库实现的可重入锁有应用任务锁、应用组件锁，重入规则为server的id。内部的锁为任务执行计划锁，内部锁为小锁，随用随放。两个外部重入锁基本已经做了很多优雅退出时和错误异常时的处理，但仍有可能存在问题，如果出现执行卡住，可以查看数据库对应锁关系是否正确，并考虑直接手工释放（目前正常不会发生，也行少部分未考虑到的极端场景可能会出现，出现后应考虑代码修复对应问题，手工释放只是临时解决）。
    * 错误处理导致代码更为复杂，大体错误处理逻辑为基于内部错误识别错误并采取对应错误处理，以及一些失败重试逻辑。
    * 执行的停止与重试功能，一者为了业务需要，另外是为了优雅退出。

* pkg/cluster：目前只是集群一般环境的定义，原本的设计是为了多集群管理设计。目前只是在pkg/app中抽象一个函数返回固定值。未来多集群可以考虑为system信息的扩展。

* **pkg/component**：各种组件定义，用于编码与解吗各种组件。面向开发提供对应的组件任务类型，实施时是框将配置转换为任务实例配置的基础。

* **pkg/graph**：将应用定义转换为执行计划的模块。执行计划将基于任务依赖关系生成图，并根据图的依赖关系，自下而上的执行。

  * 执行计划通过并发算法，通过next函数能exactly once的获取所有不相互依赖的任务节点，如果当前执行计划被无可执行节点但仍有被阻塞节点(即存在其依赖节点未标记完成的节点)，则当消费节点完成后，通过done函数将会标记完成节点，从而生成下一批可执行节点。

* **pkg/graph/task**：定义了组件任务类型，是具体执行计划的各个组件任务的执行者的实现。其中各种task还会使用到其他pkg下的模块，各种有自己的代码层级划分。这里更多的是任务分类，提供组件实例到任务实例转换的工程模式以及整体组件任务执行代码。

  * 新增任务类型需要结合pkg/component和pkg/graph/task模块编写其定义模块和执行模块。如果pkg/graph/task的具体实现比较复杂，应寻pkg其他目录编写具体实现，分割代码层级。

* pkg/helm：helm相关操作实现，如chart仓库操作、helm release操作等。如pkg/graph/task/helm.go依赖该模块实现执行helm具体操作。需要注意这里为了解决一些helm缺陷以及公司现有问题，对部分helm实现进行了魔改，有需要时应该再考虑调整。魔改部分代码位于其internal目录下

* pkg/task/log：一个简易的，基于数据库的任务执行输出日志器。

* pkg/oauth：上层应用认证包的认知鉴权实现，即hydra与as认证相关，主要用于单实例

* **pkg/sql-driver**：已有关系数据库sdk实现不能满足框架使用需求，该模块进行补充，对关系数据库进行更细粒度的分层与实现。主要是对关系数据库驱动层的分层实现。目前下分4种数据库实现，主要为错误处理统一转换为内部错误以及数据库管理接口的面向抽象接口实现。

* **pkg/store**：业务数据在持久化存储的实现，也是整个系统中代码量最多的部分。

  * mysql：关系数据库存储实现，诸如各种数据存储的持久化操作，记录了各种sql语句以及其事务的实现。
  * proton：基于proton相关系统、基础资源组件配置的存储获取，一般通过包装上文中的关系数据库存储实现部分组件的差异化处理pkg/app/executor模块中对组件数据的读写行为。该目录下的代码主要为单实例下自proton-cli-conf中读写配置的实现
  * proton/system：与上文中的proton模块类似，但主要为基于关系数据库的多实例基础资源组件配置存储。有两个用途，一为面向基础资源组件的安装/升级和管理接口，一位装饰上文的mysql模块实现，差异化在pkg/app/executor模块中对组件数据的读写行为。

* **sql-ddl**：整个服务的关系数据库的表定义，以及数据库升级模块定义。主要由cmd/server中的upgrade子命令使用。内部按数据库模块划分，每个模块划分为多种关系数据库实现。

  * 其中init目录为数据库初始化操作，一般维持最新的正确定义即可，可更改。
  * 其下pre和post目录为数据库升级模块，内容经发布后不可更改。如有数据库升级需要则按规则新增文件定义数据升级操作。文件名规则为"<日期>\_<迭代数>\_<任意说明>.yaml"。执行顺序大体为所有init，循环执行所有迭代中同一迭代的pre，然后同一迭代的post。
  * 该目录将会被当前系统状态，按日期，同一迭代数进行at lease one的方式执行，根据当前已执行日期往后执行。控制目录下文件定义不冲突，不随意调整已有文件日期，不用担心低日期文件内容的执行对产生影响高日期。
  * 具体执行逻辑可以参考pkg/store/mysql/upgrade模块

  

## 构建与开发环境搭建相关

大部分构建都是基于tag和分支管理的。由于项目管理了多个对外产物，因此代码库的一般tag为统一tag。独立tag可通过<制品名>-<tag>的方式独立设置。例如builder-v1.0.0将会得到一个builder:v1.0.0的镜像。

构建相关代码基本位于build目录下，可以结合以下介绍、脚本名和azure-pipeline.yaml文件进行理解，便于搭建开发环境、本地调试以及产品发布等。

* build/docker：镜像构建相关dockerfile
* build/script：自动脚本，如构建二进制制品，构建镜像等。基本可以通过脚本名字区分。其中meta.sh为大多数脚本需要的元数据加载脚本。
  * 日常开发使用的auto_test.sh为运行整体测试用例生成测试报告的脚本，需要配置相关环境变量并在proton环境上运行，因为如数据库操作之类打桩意义不大，该部分在逻辑部分也有，但最主要的是整体运行时是需要与实际进行自动测试的。因此部分测试用例会直接对接真实的数据库和proton的一些组件，以达到快速检测兼容性的目的。局部测试直接运行对应模块测试即可。
  * build_doc.sh脚本用于更新数据结构或者api后更新接口文档。
  * build_test_link：搭建开发环境时运行该脚本，用于将数据库测试文件链接到特定路径，使得该一个文件，全修改的目的。（其实可以再调整，目前主要是为了偷懒）


# 开放规范

项目参与者必须遵循的一些规范

1. 所有错误使用*trait.Error 表示，而不是使用 golang 标准的 error。使用的底层库返回的 error 必须包装未*trait.Error
2. \*trait.Error.Code 必须来自于代码目录 error/codes 目录及其子目录下
3. error/codes 层级目录及其子目录部分代码不允许手工编写,错误码定义流程如下。
   1. 根据模块划分或业务划分，在 error/codes 下创建或复用已有子目录
   2. 在第一步创建的目录中在 code.json 中定义错误码
   3. 运行`go run tools/generate_errors/main.go  error/codes`生成错误码
   4. 从需求分支合并代码到主线时，必须注意 error/codes/error_codes.sqlite3 文件内容的是否冲突问题
      1. 即合并代码前合并主线
      2. 重新生成错误码
      3. 对比错误码数值是否与期望相同(**可能由于其他人在其他分支的提交,导致错误码值被其他人使用**)
      4. 与期望不符时以主线为准
4. api/rest 层级目录及其子目录接口文档为代码生成,编写接口时记得完善备注。接口文档完成后，运行`build/script/build_doc.sh`生成新接口文档
   1. 暂不支持手工修改生成的接口文档
5. 合并主线前必须合并主线代码后执行`build/script/auto_test.sh`运行完整测试用例且通过
   1. 其中数据库测试用例非打桩的 UT 用例，因此需要配置数据库连接信息，详情见脚本参数。

# 开发与发布

1. 项目采用 git tag + git commit 保证唯一版本号。git tag 为语义版本号,开发过程中不增加。发布时按语义化版本号变更。
2. 发布必须使用唯一版本号。
3. 需求分支开发可使用分支便捷版本号,一般开发为本地开发。
